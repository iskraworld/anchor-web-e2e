# QA 문서 생성 프롬프트 — Fully AI 작성용

> 기능명세서 + 정책서 + Figma를 투입해 e2e-v2 자동화에 1:1 매칭되는 QA 테스트케이스 문서를 생성하는 프롬프트 템플릿.
> 신규 서비스(사주톡 등) 적용 시 모듈 코드와 역할 코드만 교체.

---

## 사용 방법

1. 아래 프롬프트 본문을 복사
2. `<...>` 표시된 부분을 실제 자료로 교체
3. AI에 투입 → `docs/qa/QA_<MODULE>_<NAME>.md` 형식으로 출력
4. 출력물을 phase0-prerequisites 체크리스트에 통과시킨 후 phase1로 진입

````
[1] TC-ID 형식: {모듈}-{섹션}-{번호} (예: TA-1-09)
   - 섹션 0 = 접근 권한, 1 = 첫 화면, 2 = 두 번째 화면

[2] 자동화 카테고리 마커 (TC-ID 옆에 붙임):
   - [D] 삭제됨 — TC를 ~~취소선~~ 처리, 비고에 "삭제 사유" 명시
   - [B] BLOCKED — UI 미출시 또는 docs 모호. 비고에 사유 명시
   - [M] 수동 검증 필요 — PG 실거래/SMS 인증/이메일 도착 등
   - [S] 권한 스킵 — 정책상 자동화 제외 (예: 회원 탈퇴)
   - 마커 없음 = 일반 자동화 가능

[3] 기대 결과 작성 — 검증 가능 단언만:
   ✅ "URL이 /search/tax-experts/123 으로 변경"
   ✅ "테이블 행이 10개 표시"
   ✅ "data-testid='firm-card' 1개 이상 visible"
   ✅ "에러 메시지 '잘못된 비밀번호' 표시"

   ❌ "적절히 표시된다"
   ❌ "자연스럽게 동작한다"
   ❌ "사용자에게 좋게 보인다"
   ❌ "충분히 노출된다"
   ❌ "결과가 표시된다" (어떤 결과? 무엇으로 식별?)

[4] 모호함을 만나면:
   - 추론 가능 → 가장 가능성 높은 해석을 기대 결과로 적고 비고에 "AI 해석 (신뢰도 X%)" 표시
   - 추론 불가 → [B] 처리하고 비고에 "docs 모호 — 명확화 필요"

[5] Figma에서 추출할 정보:
   - data-testid 추정 (Figma 컴포넌트명 기반)
   - 정확한 버튼/링크 텍스트
   - URL 패턴 (라우팅 다이어그램 기반)
   - 페이지네이션·필터·드롭다운 옵션 수
   기대 결과에 이 정보를 최대한 구체화해서 반영.

[6] 엣지케이스 강제 항목 (해당 화면마다):
   - 빈 상태 (데이터 0건)
   - 권한 차단 (해당 권한 없는 사용자)
   - 입력 검증 실패 (잘못된 값)
   - 네트워크 에러 (가능하면 [M], 아니면 생략)

[7] 도메인 정답 양방향 데이터 — 배지/태그/속성 검증 케이스:
   "X가 표시된다"만 적으면 페이지 어디 텍스트만 확인하는 fake-pass 발생.
   대신 **있어야 하는 데이터 + 없어야 하는 데이터** 둘 다 명시:

   ✅ 좋은 예시:
   - 비고: "TOP10 보유 회사: 가온세무법인 / TOP10 미보유 회사: 가감세무법인"
   - 기대 결과: "가온세무법인 카드에 TOP10 배지 노출, 가감세무법인 카드에 미노출"

   ❌ 나쁜 예시:
   - 기대 결과: "TOP10 배지가 표시된다"  ← 어떤 회사? 어떻게 식별?

   해당 데이터 없이는 자동화 불가능 → docs 보강 안 되면 [B] BLOCKED 사유: "QA 시트 부족 — 도메인 정답 비교 데이터 없음"

[8] End-to-end action chain 명시 — 액션 단계별 모두 적기:
   "X 클릭 → Y 발생" 한 줄로 끝내지 말고 단계 분해.

   ✅ 좋은 예시:
   - 테스트 절차:
     1. 주소 변경 모달 열기
     2. 도로명주소 검색 입력
     3. 결과 선택
     4. 변경 버튼 클릭
   - 기대 결과:
     - 모달이 닫힌다
     - 페이지에 변경된 주소가 반영된다

   ❌ 나쁜 예시:
   - 기대 결과: "주소가 변경된다"  ← 어디까지가 검증 범위?

   automation-patterns.md §11 참고.

[9] 사용자 동작 vs URL 직접 진입 명시:
   "GNB > X 클릭"인지 "직접 URL 진입"인지 docs에 명확히.

   ✅ 좋은 예시:
   - 테스트 절차: "홈에서 GNB > 세무 이력 관리 메뉴 클릭"
   - 기대 결과: "/tax-history-management 으로 이동, 기본 정보 탭 활성"

   ❌ 나쁜 예시:
   - 테스트 절차: "세무 이력 관리 페이지 진입"  ← page.goto 편법 유도

   automation-patterns.md §12 참고.

[10] 모호 동사 정량 차단 (자동 검출 + 재생성 강제):
   다음 12개 동사가 "기대 결과" 셀에 **단독** (수식어 없이) 사용되면 출력 거부:

   금지 동사 풀:
   표시된다 / 노출된다 / 반영된다 / 변경된다 / 동작한다 / 활성화된다 /
   펼쳐진다 / 닫힌다 / 이동한다 / 정렬된다 / 확인된다 / 적용된다

   이 동사들은 반드시 "무엇이 / 어디에 / 어떻게" 수식어와 결합:

   ✅ "data-testid='member-list' 안에 검색어 매칭 행만 표시"
   ✅ "URL이 /search?q=홍길동 으로 변경"
   ✅ "<dialog data-testid='address-confirm'> 닫힘 + 페이지의 주소 텍스트 = 입력한 도로명"
   ✅ "memberRows.count() === 1 (검색 결과 단일)"

   ❌ "결과가 표시된다"             — 어떤 결과?
   ❌ "리스트가 변경된다"           — 어떻게?
   ❌ "팝업이 닫힌다"               — 어떤 팝업?
   ❌ "메뉴가 펼쳐진다"             — 어떤 메뉴? 펼쳐짐 어떻게 식별?
   ❌ "최근 조회 순서대로 정렬된다" — 정렬 기준 데이터?

   자가 검증 단계에서 regex로 검사: 위 동사 + (마침표|줄바꿈|쉼표) 패턴 매치 시 출력 거부.

[11] 빈 기대 결과 셀 절대 금지:
   "기대 결과" 컬럼이 빈 셀이 하나라도 있으면 출력 거부.
   빈 셀은 "추론 가능" 신호가 아니라 "docs 부재" 신호 — 항상 [B] 마크:

   ✅ 처리 방식:
   - 도메인 정답 양방향 데이터 없음 → [B] + 비고 "QA 시트 부족 — 도메인 정답 비교 데이터 없음"
   - 사전 데이터 명세 없음     → [B] + 비고 "QA 시트 부족 — 사전 데이터 명세 없음"
   - 명세서 §X.Y 동작 미정     → [B] + 비고 "docs 부족 — 명세서 §X.Y 명확화 필요"

   ❌ 빈 셀 통과 (절대 금지)
   ❌ "(기대 결과 없음)" 같은 placeholder 통과 (절대 금지)

[12] TC별 사전조건 데이터 — 추상 표현 금지:
   공통 사전조건 외에 TC별 사전 데이터는 "사전조건" 컬럼에 구체 수량/이름/ID 명시.

   ✅ 좋은 예시:
   - "전체 멤버 3명 이상, 그 중 한 명 이름 = '홍길동'"
   - "현직 공무원 조회 이력 1건 이상, 조회 대상 공무원 ID = OFFICER-123"
   - "검색어 'TOP10' 매칭 결과 1건 이상 (회사명 = '가온세무법인')"

   ❌ 추상 표현 금지:
   - "현직 공무원 조회 이력"        — 몇 건? 누구?
   - "멤버 2명 이상"                — 이름은?
   - "구독 상태"                    — Team/Pro/Free 중?

   데이터 시드 방법이 명세서에 없으면 [B] BLOCKED + 비고 "QA 시트 부족 — 사전 데이터 시드 미정".

[13] 자동화 가능성 사전 평가 — TC 작성 직전 체크:
   다음 중 하나라도 해당하면 즉시 [B] 또는 [M]:

   storage state 의존 (계정 상태 사전 세팅 필요):
   - "Team 구독 상태", "구독 취소 상태", "미구독 상태", "권한 회수 후"
   - 명세서에 sign-up/구독 flow 없으면 → [B]
   - 있으면 → 사전조건 컬럼에 storage state 파일 경로 명시

   외부 시스템 의존:
   - 이메일/SMS/PG 발송 → [M]
   - 외부 API 콜 → 명세서에 fixture 없으면 [B]

   staging 동작 변동:
   - "탭 클릭이 새창/탭전환인지" 명세서에 없음 → [B]
   - "팝업 노출 조건" 시간/세션 의존 → [B]

   시간 의존:
   - "구독 만료 후", "N일 경과 후" → fixture/시간 조작 가능성 명시 안 되면 [M]

[14] 닫힌 메뉴/모달/드롭다운 — 활성화 단계 명시:
   GNB/모달/드롭다운/탭이 평소 닫혀 있고 트리거로 열리는 UI는 "활성화" 단계를 절차에 명시.

   ✅ 좋은 예시:
   - 테스트 절차:
     1. 햄버거 아이콘(data-testid='gnb-toggle') 클릭
     2. GNB 패널 열림 (data-testid='gnb-panel' visible)
     3. "세무 이력 관리" 메뉴 클릭
   - 기대 결과: /tax-history-management 으로 이동

   ❌ 나쁜 예시:
   - 테스트 절차: "GNB > 세무 이력 관리 클릭"
     ← GNB가 열려 있는지 닫혀 있는지 모호 → fake-pass 통로

   automation-patterns.md §13 참고.

정성 검증:
- [ ] 모든 TC가 "이 단언이 실패하면 명확히 어떻게 알 수 있는가?"에 답할 수 있는가
- [ ] 권한 섹션, 정상/데이터/엣지 3분류 모두 채움
- [ ] [D]/[B]/[M]/[S] 마커가 명확한 사유와 함께 있음
- [ ] data-testid 추정값 또는 정확한 텍스트가 기대 결과에 포함됨
- [ ] 배지/태그/속성 검증 TC에 도메인 정답 양방향 데이터 명시됨 ([7])
- [ ] 액션 흐름 TC가 단계별 분해 + 마지막 단계 결과 명시됨 ([8])
- [ ] navigation TC에 "X 클릭" vs "URL 직접" 구분 명시됨 ([9])
- [ ] 닫힌 메뉴/모달/드롭다운 사용 TC에 활성화 단계 명시됨 ([14])

정량 검증 (regex로 자동 검사 — 실패 시 출력 거부 + 재생성):
- [ ] 모호 어휘 0회: "적절히|자연스럽게|충분히|좋게|원활히|매끄럽게"
- [ ] 단독 모호 동사 0개 — "기대 결과" 셀에서 다음 동사가 수식어 없이 단독 사용 ([10]):
       표시된다|노출된다|반영된다|변경된다|동작한다|활성화된다|
       펼쳐진다|닫힌다|이동한다|정렬된다|확인된다|적용된다
       (각 동사 앞뒤로 "무엇이/어디에/어떻게" 수식어 필수)
- [ ] "기대 결과" 빈 셀 0개 — 빈 셀은 [B] 마크 + 비고 사유 필수 ([11])
- [ ] 사전 데이터 의존 TC의 "사전조건" 컬럼에 추상 표현 0개 ([12]):
       금지 패턴: "N명 이상" / "이력" / "조회 결과" 등 수량/ID 누락
- [ ] storage state / 외부 시스템 / staging 동작 의존 TC에 [B] 또는 [M] 마크 명시 ([13])

이 체크리스트를 통과하지 못하면 출력 금지. 통과시켜라.

위반 시 처리 순서:
1. regex 검사 실패 항목 식별
2. 해당 셀 재작성 (수식어 추가, 데이터 명시, [B] 마크 등)
3. 재검사 → 통과까지 반복
4. 그래도 통과 못 하면 → 해당 TC를 [B] BLOCKED + "docs 부족 — 명확화 필요" 비고로 종료
````

## 왜 이 프롬프트인가 — 설계 근거

### 0. 154건 AMBIGUOUS_DOC 분석 결과 (2026-05-06 v4 보강)
anchor v1에서 발생한 154건 AMBIGUOUS_DOC 마크를 카테고리별로 분석한 결과:

| 패턴 | 건수 | v4에서 차단하는 룰 |
| --- | ---: | --- |
| docs 인용 표현 모호 (표시된다/노출된다 등) | 87 | [10] 모호 동사 정량 차단 |
| docs 비어있음 (빈 셀) | 12 | [11] 빈 셀 절대 금지 |
| 사전 데이터/픽스처 부족 | 16 | [12] TC별 사전조건 데이터 강제 |
| 환경/storage state 의존 | 5 | [13] 자동화 가능성 사전 평가 |
| UI 동작 명세 모호 | 3 | [14] 닫힌 메뉴/모달 활성화 명시 |
| 기타 | 31 | [10][11][12] 조합으로 차단 |

v4에서 추가한 정량 검증(regex 자가 검사)이 87건 quote 패턴을 직접 차단하고, [11] 빈 셀 강제 [B]가 12건 empty 패턴을 차단. 결과적으로 anchor v2 docs 생성 시 AMBIGUOUS_DOC 비율이 154 → ~30 (약 80% 감소) 예상.

### 1. Fake PASS 차단을 prompt 단계에서 시작
e2e-v2의 audit가 잡는 fake-pass 패턴(`expect(body).toBeVisible()`만 있는 단언)은 **QA 문서가 모호하면 어쩔 수 없이 발생**한다. AI에게 모호한 docs를 주면 AI는 "안전한 통과" 단언을 만든다. 따라서 docs 단계에서 "검증 가능한 단언"을 강제해야 한다.

### 2. AMBIGUOUS_DOC 자동 분류 흐름과 정합
phase2-code-generation.md §모호한 docs 처리에서 정의한 3분류(추론 가능 / 추론 불가 / 약한 단언)와 동일한 분기를 prompt 단계에서 미리 적용. 결과:
- docs 작성 시점에 이미 모호 항목이 [B] 또는 "AI 해석 (신뢰도 X%)"로 마킹됨
- phase2 코드 생성이 더 매끄럽게 진행

### 3. Figma 활용으로 selector 힌트 강제
이번 세션에서 발견한 selectOption 타임아웃 같은 패턴은 docs에 selector 힌트가 없어서 "일반 selector 추측"에 의존하다 실패한 것이 많았다. Figma 컴포넌트명 → data-testid 추정 단계를 prompt에 명시해 코드 생성 시 추측 의존도 감소.

### 4. 카테고리 마커를 prompt 단계에서 사전 분류
phase1-qa-analysis가 하는 일을 docs 작성 시점에 미리 수행. phase1은 "이미 분류된 마커 검증" 만 하면 됨 → 작업 시간 단축.
