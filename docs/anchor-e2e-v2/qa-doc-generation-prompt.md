# QA 문서 생성 프롬프트 — Fully AI 작성용

> 기능명세서 + 정책서 + Figma를 투입해 e2e-v2 자동화에 1:1 매칭되는 QA 테스트케이스 문서를 생성하는 프롬프트 템플릿.
> 신규 서비스(사주톡 등) 적용 시 모듈 코드와 역할 코드만 교체.

---

## 사용 방법

1. 아래 프롬프트 본문을 복사
2. `<...>` 표시된 부분을 실제 자료로 교체
3. AI에 투입 → `docs/qa/QA_<MODULE>_<NAME>.md` 형식으로 출력
4. 출력물을 phase0-prerequisites 체크리스트에 통과시킨 후 phase1로 진입

   ✅ "data-testid='firm-card' 1개 이상 visible"
   ✅ "에러 메시지 '잘못된 비밀번호' 표시"

   ❌ "자연스럽게 동작한다"
   ❌ "사용자에게 좋게 보인다"

## 왜 이 프롬프트인가 — 설계 근거

### 1. Fake PASS 차단을 prompt 단계에서 시작
e2e-v2의 audit가 잡는 fake-pass 패턴(`expect(body).toBeVisible()`만 있는 단언)은 **QA 문서가 모호하면 어쩔 수 없이 발생**한다. AI에게 모호한 docs를 주면 AI는 "안전한 통과" 단언을 만든다. 따라서 docs 단계에서 "검증 가능한 단언"을 강제해야 한다.

### 2. AMBIGUOUS_DOC 자동 분류 흐름과 정합
phase2-code-generation.md §모호한 docs 처리에서 정의한 3분류(추론 가능 / 추론 불가 / 약한 단언)와 동일한 분기를 prompt 단계에서 미리 적용. 결과:
- docs 작성 시점에 이미 모호 항목이 [B] 또는 "AI 해석 (신뢰도 X%)"로 마킹됨
- phase2 코드 생성이 더 매끄럽게 진행

### 3. Figma 활용으로 selector 힌트 강제
이번 세션에서 발견한 selectOption 타임아웃 같은 패턴은 docs에 selector 힌트가 없어서 "일반 selector 추측"에 의존하다 실패한 것이 많았다. Figma 컴포넌트명 → data-testid 추정 단계를 prompt에 명시해 코드 생성 시 추측 의존도 감소.

### 4. 카테고리 마커를 prompt 단계에서 사전 분류
phase1-qa-analysis가 하는 일을 docs 작성 시점에 미리 수행. phase1은 "이미 분류된 마커 검증" 만 하면 됨 → 작업 시간 단축.

## 신규 서비스 적용 시

### 변경 항목 (2가지)
1. **모듈 코드**: TA/SP 등 → 신규 서비스 모듈 코드 (예: 사주톡 → HS=horoscope, UP=user-profile)
2. **역할 코드**: U2+U9 등 → 신규 서비스 역할 체계 (예: free / paid / admin)

### 그대로 재사용
- 출력 형식 (테이블 구조, 마커 규칙)
- 금지어 리스트 (적절히/자연스럽게/충분히)
- 자동 분류 키워드 (PG/SMS/탈퇴 등)
- 자가 검증 체크리스트

---

## 출력 직후 검증 절차

1. **phase0-prerequisites.md** 체크리스트로 11개 모듈(또는 신규 서비스 모듈 수) 누락 확인
2. **verify-coverage:audit** 실행 — 모호 단언 자동 검출
3. audit가 잡은 항목 → 프롬프트로 재투입해 보강 또는 [B] 마킹
4. 통과 시 phase1 → phase2 진입

## 한계 및 후속 작업

### 한계
- 기능명세서 자체가 부실하면 prompt도 한계 (모호 항목 [B] 비율 ↑)
- Figma component naming convention이 일관되지 않으면 testid 추정 정확도 저하
- 정책서·명세서 충돌 시 AI가 자체 판단 → 검토 필요

### 후속 개선 후보
- Figma → data-testid 자동 추출 도구 (별도 MCP)
- 명세서 충돌 자동 감지 (정합성 audit)
- 신규 서비스마다 "역할 코드 매핑" 사전 작성 가이드
